# 2026-06-21 开发历程

> **主题**：前端地图路线不更新问题排查 + 前端工具机制深入理解

---

## 一、问题现象

前端地图显示了当前位置和目的地，但导航后**路线不显示**（折线图没渲染）。

用户疑问："路线不是生成了折线图吗？为什么前端根本没有显示路线？"

---

## 二、问题根因分析

### 1. 直接原因

`update_map` 是前端工具（`useFrontendTool` 注册），但 **LLM 看不到这个工具**。

**完整链路分析**：
```
前端 useFrontendTool("update_map")
    ↓ CopilotKit 通过 AG-UI 协议发送
    ↓ 
后端 ag_ui_langgraph 收到 input.tools = [{name:"update_map", ...}]
    ↓
langgraph_default_merge_state() 把 tools 存入 state["tools"]
    ↓
❌ 断点：子 Agent 创建时只看 MCP tools，不看 state["tools"]
```

### 2. 根本原因

- 后端 MCP 工具列表中没有 `update_map`
- LLM 只能看到 MCP 注册的工具（`plan_route`, `search_poi` 等）
- `useFrontendTool` 注册的工具定义虽然传到了后端，但没有被注入到子 Agent 的工具列表中
- 结果：LLM 调用了 `plan_route` 拿到路线数据后，直接回复文字，不会调用 `update_map`

---

## 三、修复方案（第一版）

### 改动 1：后端添加 `update_map` MCP 工具

在 `navigation_tools.py` 中新增工具，让 LLM 能看到并调用：

```python
@mcp.tool()
def update_map(
    action: str,
    destination: str = "",
    destination_lat: float = 0,
    destination_lng: float = 0,
    distance_km: float = 0,
    duration_min: float = 0,
    steps: list[str] = [],
    route_coords: list[list[float]] = [],
    pois: list[dict] = [],
) -> str:
    """
    更新前端地图显示。在规划导航路线或搜索 POI 后必须调用此工具来同步地图。
    """
    if action == "navigate":
        return f"地图已更新：导航至 {destination}，全程 {distance_km}km，约 {duration_min} 分钟"
    elif action == "search_poi":
        count = len(pois)
        return f"地图已标记 {count} 个兴趣点"
    elif action == "clear":
        return "地图已清除导航数据"
    return "未知操作"
```

### 改动 2：导航 Agent 添加工具过滤

在 `navigation_agent.py` 中让 Agent 能看到 `update_map`：

```python
navigation_tools = [
    t for t in tools
    if any(kw in t.name for kw in ["plan_route", "search_poi", "traffic", "update_map"])
]
```

同时在 system prompt 中添加强制规则：

```
重要！地图同步规则：
- 调用 plan_route 后，必须调用 update_map(action="navigate")，把 plan_route 返回的 destination、destination_coord、distance_km、duration_min、steps、route_coords 传过去
- 调用 search_poi 后，必须调用 update_map(action="search_poi")，把 pois 列表传过去
- update_map 的 route_coords 格式为 [[lat, lng], ...]
- update_map 的 pois 格式为 [{"name": "xx", "lat": 31.23, "lng": 121.47}, ...]
```

### 改动 3：前端用 `useRenderTool` 监听工具调用

在 `App.tsx` 中替换 `useFrontendTool` 为 `useRenderTool`：

```tsx
useRenderTool({
  name: "update_map",
  parameters: z.object({
    action: z.string(),
    destination: z.string().optional(),
    // ... 其他参数
  }),
  render: ({ toolCallId, status, parameters }) => {
    if (status === "complete") {
      // 更新地图状态
      if (parameters.action === "navigate") {
        setMapState(...)
      }
    }
    return <></>; // 不渲染 UI，只触发副作用
  },
});
```

### 修复后的执行流程

```
用户说"导航到外滩"
    ↓
Supervisor → 转给 navigation_agent
    ↓
navigation_agent 调用 plan_route → 拿到路线坐标数据
    ↓
navigation_agent 调用 update_map(action="navigate", route_coords=...) → 后端执行，返回"地图已更新"
    ↓
AG-UI 流式推送 TOOL_CALL 事件给前端
    ↓
CopilotKit 渲染工具调用 → useRenderTool 检测到 update_map 完成
    ↓
setMapState() 更新地图 → 蓝色路线折线显示在地图上 ✅
```

## 四、核心疑惑：useFrontendTool vs useRenderTool

### 用户疑问

> "update_map 虽然是前端工具，不是也可以更新地图吗？也可以监听某个后端调用工具，触发它更新地图吗？为啥一定要用 useRenderTool？"

### 关键区别

| 特性 | `useFrontendTool` | `useRenderTool` |
|------|-------------------|-----------------|
| **角色** | 工具定义 + 执行器 | 工具调用监听器 |
| **工具代码执行位置** | 浏览器 | 后端服务器 |
| **LLM 能看到吗** | 前端传给 LLM（但有传递限制） | 不管（后端本来就有） |
| **谁执行工具** | 浏览器的 handler 函数 | 后端的 Python 函数 |
| **返回结果给 LLM** | handler 的 return 值 | 后端函数的 return 值 |
| **能访问浏览器 API** | ✅ 能（DOM、定位、文件） | ❌ 不能 |
| **能访问后端资源** | ❌ 不能 | ✅ 能（数据库、文件等） |

### 类比解释

```
useFrontendTool = 你在公司里注册了一个岗位叫"地图更新员"
                  只有别人喊你名字"update_map"你才干活
                  没人喊你 → 你坐着不动

useRenderTool   = 你在公司装了个监控摄像头
                  不管谁干了什么活，你都看到了
                  看到"有人导航了" → 你主动去更新地图
```

### 为什么 useFrontendTool 失败

```
1. useFrontendTool 注册了 update_map ✅
   → CopilotKit 把工具定义发给后端
   → 存在 state["tools"] 里

2. 用户说"导航到外滩"
   → Supervisor 转给 navigation_agent
   → navigation_agent 只看到 MCP 工具（plan_route, search_poi...）
   → LLM 看不到 update_map（因为子 Agent 没有拿到前端工具）

3. LLM 调用 plan_route → 拿到路线数据
   → LLM 直接生成文字回复
   → 不会调用 update_map（因为它不知道有这个工具）

4. useFrontendTool 的 handler 从未被执行
   → setMapState() 从未被调用
   → 地图不更新 ❌
```

## 五、前端工具传递方案对比

### 问题：前端工具怎么传到后端？

当前架构是 **Supervisor + 5 个子 Agent + 自定义 MCP 工具**，工具分配链路是：

```
MCP Server → load_mcp_tools() → supervisor → 子 Agent 按名字过滤
                                              ↑
                                    子 Agent 只看到过滤后的 MCP 工具
                                    看不到 state["tools"] 里的前端工具
```

### 三种解决方案

#### 方案 A：创建时传入（简单但有限制）

修改子 Agent 创建函数接受额外工具参数：

```python
def create_navigation_agent(tools: list[BaseTool], frontend_tools: list = None):
    all_tools = tools + (frontend_tools or [])
    navigation_tools = [
        t for t in all_tools
        if any(kw in t.name for kw in ["plan_route", "search_poi", "traffic", "update_map"])
    ]
```

**问题**：前端工具在 `build_supervisor_graph()` 时还不存在，要等请求进来才有。

#### 方案 B：运行时动态注入（推荐）

在 LangGraph 图里加一个预处理节点：

```python
def inject_frontend_tools_node(state: dict) -> dict:
    """从 state["tools"] 提取前端工具，动态注入到子 Agent"""
    frontend_tools = state.get("tools", [])
    if not frontend_tools:
        return state
    # 按工具名分发给对应子 Agent
    return state

graph_builder.add_node("inject_tools", inject_frontend_tools_node)
graph_builder.add_edge("__start__", "inject_tools")
graph_builder.add_edge("inject_tools", "supervisor")
```

#### 方案 C：统一管理（最干净）

```python
TOOL_ROUTING = {
    "navigation": ["plan_route", "search_poi", "traffic", "update_map"],
    "media":      ["play_music", "pause_music", "next_song", "set_volume"],
    "vehicle":    ["control_window", "set_climate", "lock_doors", "set_seat"],
    "weather":    ["get_weather", "get_forecast"],
    "reminder":   ["create_reminder", "list_reminders"],
}

def create_navigation_agent(all_tools: list):
    """all_tools = MCP工具 + 前端工具，按名字筛选"""
    my_tools = [t for t in all_tools if t.name in TOOL_ROUTING["navigation"]]
    return create_agent(model, tools=my_tools, ...)
```

### 工具需要传入每个 Agent 吗？

**当前做法（筛选）**：是的，需要筛选。

```
所有工具 (16个 MCP + N个前端)
    ↓ 按名字过滤
navigation_agent: 只拿到 4 个工具
media_agent:      只拿到 5 个工具
vehicle_agent:    只拿到 5 个工具
...
```

**为什么不全给每个 Agent**：
- 全给：每个 Agent 看到 16+N 个工具 → LLM 选择困难，容易调错工具
- 筛选：每个 Agent 只看到自己的 2-5 个工具 → LLM 选择精准

**但前端工具（如 `update_map`）比较特殊**：它需要跨域调用，可以同时给多个相关 Agent。

## 六、真实项目工具使用模式

### 方式 A：后端工具 + 前端渲染（推荐）

适合多 Agent / Supervisor 架构：

```python
# 后端：定义工具（LLM 能看到）
@mcp.tool()
def update_map(action, destination, route_coords, ...):
    return "地图已更新"
```

```tsx
// 前端：监听工具执行，做 UI 更新
useRenderTool({
  name: "update_map",
  render: ({ status, parameters }) => {
    if (status === "complete") setMapState(...)  // 副作用
    return <></>
  }
})
```

### 方式 B：前端工具拦截

适合单 Agent / CopilotKit Cloud 简单项目：

```tsx
useFrontendTool({
  name: "show_weather",
  parameters: z.object({ city: z.string() }),
  handler: async ({ city }) => {
    const data = await fetch(`/api/weather/${city}`).then(r => r.json())
    setWeatherData(data)
    return `天气：${data.temp}°C`  // 返回给 LLM 继续推理
  }
})
```

### 方式 C：纯后端 MCP 工具

适合大多数生产项目：

```python
@mcp.tool()
def search_database(query): ...

@mcp.tool()
def send_email(to, body): ...
```

前端只管聊天展示，不干预工具执行。

### 对比总结

| 方式 | 谁执行工具 | 适用场景 | Supervisor 架构能用吗 |
|------|-----------|---------|---------------------|
| `useFrontendTool` | 浏览器 | 单 Agent / CopilotKit Cloud | ❌ 不能（子 Agent 看不到） |
| `useRenderTool` + 后端工具 | 后端 | 多 Agent / Supervisor | ✅ 能（当前方案） |
| 纯后端 MCP 工具 | 后端 | 大多数生产项目 | ✅ 能 |

## 八、关键知识点总结

### 1. CopilotKit 工具传递机制

- `useFrontendTool` 注册的工具定义会通过 AG-UI 协议传给后端
- 后端 `ag_ui_langgraph` 把它们存入 `state["tools"]`
- **但在 Supervisor 架构下，子 Agent 创建时绑定固定 MCP 工具列表，看不到 `state["tools"]`**

### 2. 前端工具 vs 后端工具

- **前端工具**：代码在浏览器跑，能访问浏览器 API（DOM、定位、文件）
- **后端工具**：代码在服务器跑，能访问后端资源（数据库、文件、网络）

### 3. useRenderTool 的本质

- 它**不是工具定义**，而是**工具调用监听器**
- 不会让 LLM 知道这个工具的存在
- 只在前端监听后端工具的执行结果
- 必须配合后端工具一起用

### 4. Supervisor 架构的工具分配

```
MCP 工具：按领域严格筛选 ✅
前端工具：可以同时给多个相关 Agent ✅
```
