  ↓ 请求体里带上 copilotkit.actions = [update_map, select_origin] 的 schema
Supervisor 识别意图 → 路由到 navigation_agent
  ↓ navigation_agent 的 CopilotKitMiddleware.wrap_model_call：
  ↓   把 state["copilotkit"]["actions"] 的前端工具 merge 进本次模型调用的 tools
  ↓ LLM 现在能看见 select_origin / update_map，决定先调 select_origin（缺起点）
  ↓ CopilotKitMiddleware.after_model：识别 select_origin 是前端工具，
  ↓   把它从"待后端执行"的 tool_calls 里摘出来，交给 AG-UI 推到前端

  ↓ 用户点"公司" → respond("公司") → AG-UI 把结果回传
LLM 拿到起点 → 调 plan_route(MCP 高德工具) → 再调 update_map(前端工具)
前端 useFrontendTool.handler 更新地图
  ↓ navigation_agent 汇总 → Supervisor → 流式回复用户

5. P0：统一到 create_agent

改法：4 个子 Agent 从 create_react_agent → create_agent
以 media_agent.py 为例（vehicle/weather/reminder 同理）：
# ❌ 当前（已弃用 API）
from langgraph.prebuilt import create_react_agent

def create_media_agent(tools: list[BaseTool]):
    media_tools = [t for t in tools if any(kw in t.name for kw in ["music", "song", "volume", "playlist"])]
    return create_react_agent(
        model=create_llm(),
        tools=media_tools,
        name="media_agent",
        prompt=MEDIA_PROMPT,          # 注意：create_react_agent 用 prompt
    )

# ✅ 改为（LangChain v1 标准）
from langchain.agents import create_agent

def create_media_agent(tools: list[BaseTool]):
    media_tools = [t for t in tools if any(kw in t.name for kw in ["music", "song", "volume", "playlist"])]
    return create_agent(
        model=create_llm(temperature=0.3),
        tools=media_tools,
        name="media_agent",
        system_prompt=MEDIA_PROMPT,   # 注意：create_agent 用 system_prompt（不是 prompt）

两个易踩点：
1. 参数名从 prompt 改成 system_prompt。
2. 这 4 个 Agent 不需要 CopilotKitMiddleware（它们不用前端工具）。只有 navigation_agent 需要，保持它现有的 middleware=[CopilotKitMiddleware()] 即可。
navigation_agent.py 基本不用动，它已经是正确写法。唯一建议：把 create_llm() 改成 create_llm(temperature=0.3)（配合第 8 节修复 lru_cache 后温度才会生效）。

6. P0：让前端工具真正传到子 Agent
6.1 CopilotKitMiddleware 到底做了什么（读源码得出）
from copilotkit import CopilotKitMiddleware 是一个真正的 LangChain v1 AgentMiddleware，它在 Agent 循环里挂了几个钩子：
钩子	作用
wrap_model_call	读 state["copilotkit"]["actions"]（前端传来的工具），merged_tools = [*request.tools, *frontend_tools] 注入本次模型调用 → 让 LLM 看见前端工具
after_model	LLM 调了前端工具后，把这些调用从"待后端 ToolNode 执行"的列表里摘出来（后端不执行前端工具），暂存
after_agent	把摘出来的前端工具调用还原到消息里，交给 AG-UI 推送到前端执行
before_agent	注入前端 App Context
所以不需要 stub、不需要手写 bind_tools——只要：
1. 子 Agent 用了 CopilotKitMiddleware（navigation 已用 ✅）
2. state 里确实有 copilotkit.actions 这个字段，且能传到子 Agent ⬅️ 这就是 #2 的坑
6.2 真正的坑：Supervisor 没声明 copilotkit 状态通道
build_supervisor_graph() 里：
# ❌ 当前：没传 state_schema，Supervisor 用默认状态（不含 copilotkit 通道）
supervisor = create_supervisor(
    agents=agents,
    model=create_llm(temperature=0.3),
    prompt=_build_prompt,
)
AG-UI 端点把前端工具放进图输入的 copilotkit.actions。但如果 Supervisor 父图的 state_schema 里没有 copilotkit 这个通道，这个字段在父图入口就被丢弃，根本传不到 navigation 子图的 CopilotKitMiddleware。
# ✅ 修复：显式传入继承自 CopilotKitState 的状态，保证 copilotkit 通道贯通父图→子图
from app.graph.state import AutoMindState

supervisor = create_supervisor(
    agents=agents,
    model=create_llm(temperature=0.3),
    prompt=_build_prompt,
    state_schema=AutoMindState,   # ← 关键：让 copilotkit.actions 能传到子 Agent
)
AutoMindState(CopilotKitState) 已带 copilotkit 通道（见 state.py）。这样链路才完整：
AG-UI 端点 → Supervisor(AutoMindState) → navigation 子图 → CopilotKitMiddleware.wrap_model_call → LLM 看见前端工具。
6.3 自检：三处工具名必须逐字符一致
位置	名称	说明
App.tsx 的 useFrontendTool({ name })	"update_map"	前端 handler 注册名
App.tsx 的 useHumanInTheLoop({ name })	"select_origin"	前端 HITL 名
Prompt 里引导 LLM 调用的名字	update_map / select_origin	必须一致
注意：用 CopilotKitMiddleware 后后端不需要再写同名 stub（前端工具由 state["copilotkit"]["actions"] 动态注入）。这点与早期文档的"stub 方案"相反——以本节为准。

7. P1：Human-in-the-Loop 的两条官方路线
LangChain v1 + CopilotKit 下，HITL 有两条都正确的路线，按场景选：
路线 A：前端驱动（useHumanInTheLoop）——本项目 select_origin 用这个
适合"前端弹 UI 让用户点选/补充输入"。整个暂停/渲染/回传由 CopilotKit + AG-UI 在前端闭环，后端只要：
1. navigation_agent 带 CopilotKitMiddleware（已具备）
2. Supervisor 传 state_schema=AutoMindState（第 6.2 修）
前端（App.tsx，当前写法基本可用）：
import { useHumanInTheLoop, ToolCallStatus } from "@copilotkit/react-core/v2";
import { z } from "zod";

useHumanInTheLoop({
  name: "select_origin",                       // 与 prompt 中的工具名一致
  description: "让用户选择导航起点位置。",
  parameters: z.object({
    options: z.array(z.string()).describe("可选起点列表"),
    message: z.string().describe("提示文本"),
  }),
  render: ({ args, status, respond }) => {
    if (status === ToolCallStatus.Executing && respond) {
      const options = (args as any)?.options ?? ["家", "公司", "火车站", "机场"];
      return (
        <div className="grid grid-cols-2 gap-2">
          {options.map((opt: string) => (
            <button key={opt} onClick={() => respond(opt)}>{opt}</button>
          ))}
        </div>
      );
    }
    return <div className="text-green-400">✓ 起点已选择</div>;
  },
});
关于 agentId：当前代码写了 useHumanInTheLoop({ agentId: "navigation_agent", … })。
v2 的 useHumanInTheLoop 是按工具名匹配的。agentId 用于多 agent 场景下指定目标 agent；本项目是单一 AG-UI agent（automind），子 Agent 是后端内部细节，前端并不直接看到 navigation_agent。
建议先去掉 agentId 验证（按工具名匹配最稳）；若你确实接了多 agent 注册再按 useHumanInTheLoop 参考填正确的 agentId。
路线 B：后端驱动审批（HumanInTheLoopMiddleware）——给"危险操作"用
适合"后端流程必须人工审批"，比如行驶中解锁车门、扣费等。这是 LangChain 官方 HITL，写在 create_agent 的 middleware 里：
from langchain.agents import create_agent
from langchain.agents.middleware import HumanInTheLoopMiddleware

def create_vehicle_agent(tools: list[BaseTool]):
    vehicle_tools = [t for t in tools if any(kw in t.name for kw in ["window", "climate", "door", "seat", "vehicle"])]
    return create_agent(
        model=create_llm(temperature=0.3),
        tools=vehicle_tools,
        name="vehicle_agent",
        system_prompt=VEHICLE_PROMPT,
        middleware=[
            HumanInTheLoopMiddleware(
                interrupt_on={
                    "lock_doors": True,                                   # 需要审批
                    "control_window": {"allowed_decisions": ["approve", "reject"]},
                    "get_vehicle_status": False,                          # 只读，免审批
                },
                description_prefix="⚠️ 该车辆操作需要确认",
            ),
        ],
    )
恢复执行（决策类型 approve / edit / reject / respond）：
from langgraph.types import Command
agent.invoke(Command(resume={"decisions": [{"type": "approve"}]}), config=config, version="v2")
硬性要求（官方）：HumanInTheLoopMiddleware 必须配 checkpointer（见第 9 节），否则 interrupt 无法保存/恢复状态。
❌ 仍然不要做：把 interrupt() 放进 MCP 工具
# ❌ 必崩：MCP Server 是 stdio 独立子进程，没有图执行上下文 / checkpointer
@mcp.tool()
async def select_origin(...):
    return interrupt({...})   # 找不到图上下文，抛异常或永久阻塞
langgraph.types.interrupt() 只能在 LangGraph 图节点内调用（HumanInTheLoopMiddleware 内部正是这么做的）。这条早期文档说对了，继续保留。
一句话选型
场景	用
前端弹卡片让用户点选/输入（select_origin）	路线 A：useHumanInTheLoop（前端）
后端流程要人工批准危险操作（解锁、扣费）	路线 B：HumanInTheLoopMiddleware（后端，需 checkpointer）
MCP 工具里 interrupt()	❌ 永远不要

8. P1：LLM 工厂 lru_cache 温度 Bug
models/llm.py 用 @lru_cache(maxsize=1)：第一次调用（默认 temperature=0.7）后实例被缓存，之后 create_llm(temperature=0.3) 因为 maxsize=1 仍可能命中旧缓存键/复用同实例，温度参数形同虚设。
# ❌ 当前
@lru_cache(maxsize=1)
def create_llm(model: str | None = None, temperature: float = 0.7, streaming: bool = True) -> ChatOpenAI:
    ...

# ✅ 改为按参数缓存（不同 (model, temperature, streaming) 各缓存一份）
from functools import lru_cache

@lru_cache(maxsize=8)
def create_llm(model: str | None = None, temperature: float = 0.7, streaming: bool = True) -> ChatOpenAI:
    return ChatOpenAI(
        model=model or settings.LLM_MODEL,
        temperature=temperature,
        streaming=streaming,
        api_key=settings.LLM_API_KEY,
        base_url=settings.LLM_API_BASE,
    )
lru_cache 的 key 是入参元组，maxsize=8 后 create_llm(temperature=0.3) 与 create_llm(temperature=0.7) 会各自缓存，互不覆盖。

9. P1：对话持久化
MemorySaver 重启即失忆，且路线 B 的后端 HITL 需要持久 checkpointer才能跨请求恢复 interrupt。
# ❌ 当前
from langgraph.checkpoint.memory import MemorySaver
graph = supervisor.compile(checkpointer=MemorySaver())
单机持久化用 AsyncSqliteSaver。它的 from_conn_string() 返回异步上下文管理器，必须在应用生命周期内保持打开，不能 async with 后立即 return（一退出 with 连接就关）：
# backend/app/graph/supervisor.py
from contextlib import AsyncExitStack
from langgraph.checkpoint.sqlite.aio import AsyncSqliteSaver
from app.config import settings

_checkpointer_stack = AsyncExitStack()
_checkpointer = None

async def get_checkpointer():
    """进程级单例：连接在整个应用生命周期内保持打开"""
    global _checkpointer
    if _checkpointer is None:
        db_path = str(settings.SQLITE_DB_PATH).replace("memory.db", "checkpoints.db")
        _checkpointer = await _checkpointer_stack.enter_async_context(
            AsyncSqliteSaver.from_conn_string(db_path)
        )
    return _checkpointer

async def close_checkpointer():
    await _checkpointer_stack.aclose()
# build_supervisor_graph 里：
graph = supervisor.compile(checkpointer=await get_checkpointer())
# main.py 的 lifespan：yield 之后关闭
@asynccontextmanager
async def lifespan(app: FastAPI):
    ...
    yield
    from app.graph.supervisor import close_checkpointer
    await close_checkpointer()

10. P2：删除会消费请求体的调试中间件
main.py 的 debug_tools_middleware 对每个 /agent 请求 await request.body()。Starlette 的 BaseHTTPMiddleware 下游用同一 scope 重新构造请求，body 被提前读取后下游可能读到空，导致 AG-UI 端点解析失败（即便有时凑巧不崩，也是隐患 + 多一次 JSON 解析开销）。
它只是调试用，直接删掉最干净：

async def debug_tools_middleware(request: Request, call_next):
    if request.url.path == "/agent" and request.method == "POST":
        body = await request.body()   # 消费 body 的隐患
        ...
    return await call_next(request)
想看前端传了哪些工具，正确做法是在 Agent 内部打印 state["copilotkit"]["actions"]（不碰 HTTP 层）：

12. 多 Agent 架构
Q：用 langgraph-supervisor 的 Supervisor + 5 子 Agent，是最佳实践吗？
A：是一种主流且正确的实践，本项目可继续用。 但 LangChain v1 时代多了一个官方选项，了解后按需选：
方案	出处	适合	说明
create_supervisor（当前）	langgraph-supervisor	明确领域分工、专家路由	清晰、稳。本项目保留
SubAgentMiddleware + create_agent	deepagents.middleware.subagents（官方）	主 Agent 把任务拆给子 Agent 并行/隔离上下文	LangChain v1 官方"Planning & delegation"推荐，子 Agent 各自独立上下文
手写 StateGraph 路由	LangGraph	需要精细流程控制	代码量大、最灵活
单个 create_agent	LangChain	领域单一、工具少	最简单
结论：当前 Supervisor 架构没问题，先把 #1/#2 修好让它跑通。等需要"子任务并行 + 上下文隔离"时，再评估迁到 SubAgentMiddleware（官方 Agents 文档的 "Planning and delegation" 一节有完整示例）。

13. 动态工具注入能力
Q：LangGraph 现在支持运行时动态注入工具吗？
A：支持，而且本项目的前端工具就是靠它工作的。 两种方式：
1. CopilotKitMiddleware（本项目用）：wrap_model_call 钩子在每次模型调用前把 state["copilotkit"]["actions"] 注入 tools。前端这次带了什么工具，LLM 这次就能看见——天然动态，无需重编图。
2. 自定义中间件 / 自定义节点 bind_tools：若你想注入"非前端来源"的动态工具，可以写自己的 AgentMiddleware：
from langchain.agents.middleware import AgentMiddleware, ModelRequest, ModelResponse

class DynamicToolsMiddleware(AgentMiddleware):
    def wrap_model_call(self, request: ModelRequest, handler):
        extra = build_tools_from_somewhere(request.state)   # 运行时算出工具
        return handler(request.override(tools=[*request.tools, *extra]))
LangChain v1 的 middleware 体系（wrap_model_call / before_model / after_model / wrap_tool_call 等钩子）就是官方的"运行时定制 Agent"机制。CopilotKitMiddleware、HumanInTheLoopMiddleware 都是它的实例。

15. 亮点功能扩展路线图
下面每个功能都标注"从哪入手 + 具体怎么做"，小白可按图施工。
亮点 1：危险操作二次确认（最快出彩，复用官方 HITL）
● 入手点：vehicle_agent.py
● 做法：按第 7 节"路线 B"，给 lock_doors、control_window 加 HumanInTheLoopMiddleware。前端用 useInterrupt 渲染确认弹窗。
● 效果：说"打开车门"时，车机弹"确认解锁？"，体现安全设计。
亮点 2：流式语音播报（TTS）
● 入手点：前端 App.tsx 订阅 AG-UI 事件 + 后端不动
● 做法：用 useAgent() 的 agent.subscribe({ onTextMessageContentEvent })（见官方 "Connect AG-UI agents"）拿到流式文本，逐句喂给浏览器 SpeechSynthesis 或云 TTS。
● 效果：助手边想边说，贴近真车机体验。
亮点 3：主动关怀（Proactive）
● 入手点：graph/supervisor.py 的 _build_prompt 已注入 pending_reminders/天气
● 做法：加一个定时任务（FastAPI BackgroundTasks 或 APScheduler），命中条件时通过 AG-UI 主动 push 一条消息（如"前方 2km 拥堵，是否重新规划？"）。
● 效果：从"被动应答"升级到"主动建议"。
亮点 4：生成式 UI 卡片（Generative UI）
● 入手点：前端 useRenderTool / useComponent（CopilotKit v2）
● 做法：给 plan_route 结果绑一个路线卡片组件、给天气绑天气卡片。LLM 调用工具时自动渲染富 UI，而非纯文字。
● 效果：聊天区出现可交互卡片，质感大幅提升。参考官方 "Generative UI / Tool Rendering"。
亮点 5：长期记忆个性化
● 入手点：已有 ChromaDB + SQLite 记忆模块
● 做法：在 reminder_agent 的 save_user_preference 里沉淀偏好（常去地点、爱听的歌），_build_prompt 召回后影响推荐。
● 效果："导航回家"无需问起点、"放点音乐"直接放偏好歌单。
亮点 6：多模态（车况图像/语音输入）
