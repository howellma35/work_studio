#### 如何读 Langfuse Trace 树

Langfuse Trace 是一棵树形结构，每个节点是一个 **Observation（观测点）**，有如下几类：

| 类型 | 含义 | 对应代码 |
|------|------|---------|
| `CHAIN` | 可组合的调用链 | LangChain Runnable、LangGraph 节点 |
| `AGENT` | Agent 执行单元 | LangGraph Sub-agent、React Agent |
| `GENERATION` | 一次 LLM 调用 | ChatOpenAI、ChatQwen 等 |
| `TOOL` | 工具调用 | LangChain Tool 执行 |

每个节点有 `level` 字段：`DEFAULT`（正常）或 `ERROR`（异常），用红色标出。

#### 排查时的关键观察点

**观察点 1：定位「第一个 ERROR 节点」**  
在 Langfuse Trace 树中，优先找 `level=ERROR` 的最深节点（叶子节点）。本例中最深的 ERROR 是 `call_model [CHAIN, ERROR]`，说明问题出在 LLM 调用前的消息准备阶段，不是 LLM 本身的错误。

**观察点 2：找到 ERROR 节点的错误详情**

Langfuse Trace 详情页的布局：左侧是时间轴树，点击某个节点后右侧展开该节点的详情，详情区域有三个 Tab：**Input / Output / Metadata**。

**ERROR 信息在哪里？**

Langfuse UI 本身**不会**把 `statusMessage` 单独显示为一个明显的字段——它在 UI 上的体现只是时间轴里节点名称旁边的**红色小圆点**。要看具体错误内容，有两个办法：

**办法 1（推荐）：看 Metadata Tab 里的 `statusMessage`**

点击红色节点（如 `call_model`）→ 右侧切换到 **Metadata** Tab →  
里面是一个 JSON 对象，找 `statusMessage` 字段，内容就是完整的异常文本：

```json
{
  "statusMessage": "Found AIMessages with tool_calls that do not have a corresponding ToolMessage. Here are the first few of those tool calls: [{'name': 'select_origin', ...}]"
}
```

> 如果 Metadata Tab 里没有 `statusMessage`（某些版本放在根层而不在 metadata 对象里），则用办法 2。

**办法 2：导出 Trace JSON 直接搜索**

在 Trace 详情页右上角找「Export」或「···」菜单 → 导出为 JSON → 在文件里全局搜索 `"level": "ERROR"` 或 `"statusMessage"`，即可找到所有报错节点的原始信息。本次排查用的 `trace.txt` 就是这样得到的，里面每个 observation 都有 `level` 和 `statusMessage` 字段可以直接看。

**时间轴上怎么识别 ERROR 节点**

- 正常节点：名称旁边没有特殊标记，或绿色
- ERROR 节点：名称旁边有**红色圆点**，点击后右侧 Metadata Tab 里有 `statusMessage`

本例中，时间轴里 `supervisor → agent → call_model` 这条链路的节点都带红点，最深的 `call_model` 是真正出错的位置。

这里的 `statusMessage` 直接告诉了「哪个 tool_call 孤立了」—— 是 `select_origin`。

**观察点 3：往上溯源，找 select_origin 是什么时候产生的**

**观察点 4：看时间轴，理解两次请求的边界**  
- 第一轮 `supervisor`：`06:38:52.374Z` → `06:38:54.250Z`（约 1.8s，正常完成）
- `navigation_agent`：`06:38:54.252Z` → `06:38:57.627Z`（约 3.4s，正常完成）

## 三、根本原因分析

### 3.1 消息历史的两个作用域

本项目采用 **Supervisor + Sub-agent** 的多 Agent 架构：

```
Supervisor 消息历史（外层）
└── navigation_agent 消息历史（子层，由 langgraph-supervisor 独立管理）
```

这两个作用域的消息历史是**独立存储的**（通过 LangGraph checkpoint 持久化）。

### 3.2 第一轮请求的正常流程

```
请求1: 「导航去陆家嘴」
  → Supervisor 路由到 navigation_agent
  → navigation_agent 的 LLM 决定调用 select_origin 前端工具
  → 产生 AIMessage { tool_calls: [select_origin(...)] }  ← 存入子层历史
  → CopilotKitMiddleware.after_model 拦截，将 select_origin 事件发送到前端
  → 前端弹出选择面板，Agent 流结束（等待用户输入）
```

此时 navigation_agent 的子层消息历史存档：
```
[HumanMessage("导航去陆家嘴"), AIMessage(tool_calls=[select_origin])]
                                                          ↑
                                              没有对应的 ToolMessage！
```

### 3.3 第二轮请求触发崩溃

```
请求2: 用户选择了「家」作为起点
  → CopilotKit / AG-UI 将完整对话历史（含用户选择结果）发给后端
  → Supervisor 再次路由到 navigation_agent
  → navigation_agent 尝试 call_model
  → LangGraph 在调用 LLM 前执行 _validate_chat_history()
  → 发现 checkpoint 里有 AIMessage(select_origin tool_call)
    但没有对应的 ToolMessage → 抛出 ValueError ❌
```

### 3.4 为什么 ToolMessage 没有注入到子层？

CopilotKit 的前端工具执行结果（ToolMessage）是通过 AG-UI 协议写回到**根级别（root）消息历史**中的。

但 `langgraph-supervisor` 为每个子 Agent 维护了**独立的消息作用域**（通过 `state["messages"]` 的 scoped 切片）。当前端工具结果返回时：

- ✅ 根级消息历史：包含 select_origin 的 ToolMessage  
- ❌ navigation_agent 子层历史：**仍然缺少** ToolMessage

这是 `langgraph-supervisor` 的消息隔离机制与 CopilotKit 前端工具注入机制之间的**架构兼容性问题**。

## 三补、关键澄清：useHumanInTheLoop 不是「不支持」，而是「放错了层」

> 先纠正一个容易误导的结论：**前端用 `useHumanInTheLoop` 完全正确，CopilotKit 官方就是这么做的，不需要改回后端定义工具。** 真正的问题不是「该不该用 HITL」，而是「HITL 工具被挂在了 `langgraph-supervisor` 的子 Agent 上」。

### 为什么 `update_map` 不报错，偏偏 `select_origin` 报错？

两个都是前端工具，区别在于**是否跨请求暂停**：

| 工具 | 前端 Hook | 是否暂停 | 结果 |
|------|-----------|---------|------|
| `update_map` | `useFrontendTool` | ❌ 不暂停，**同一个 run 内**立即执行 handler 并返回 ToolMessage | tool_call 和 ToolMessage 成对出现，校验通过 ✅ |
| `select_origin` | `useHumanInTheLoop` | ✅ **暂停整个 run**，等用户在前端选择，下一个请求才带回结果 | 第一个 run 结束时只有 tool_call、没有 ToolMessage，留下「孤儿」 ❌ |

**核心机制差异**：`useFrontendTool` 在当前 run 内闭环；`useHumanInTheLoop` 会让 run 在 tool_call 处中止，把控制权交给前端，**结果在下一次请求才回来**。一旦这个「悬而未决的 tool_call」挂在 `langgraph-supervisor` 子 Agent 上，并随着子 Agent 消息冒泡进入 supervisor 的共享历史，下一轮 supervisor 重新跑 `call_model` 时就会校验失败。

### 官方文档怎么说（CopilotKit Sub-Agents 指南）

CopilotKit 官方的 **Sub-Agents** 文档里有一句关键注释（直接引用）：

> "We intentionally do **NOT** attach the full `CopilotKitMiddleware` here [on the sub-agents] — the supervisor handles App-Context / frontend-tool injection for the whole run, and adding it on sub-agents would double-inject prompt state."

也就是说，**官方推荐的多 Agent 写法是：`CopilotKitMiddleware`（以及前端工具 / HITL）挂在 supervisor 上，而不是挂在子 Agent 上。** 本项目恰好相反——把 `CopilotKitMiddleware()` 挂在了 `navigation_agent` 子 Agent 上，HITL 的 `select_origin` 因此发生在子 Agent 内部，这正是冲突的根源。

此外，官方的「supervisor + 子 Agent」并不是用 `langgraph-supervisor` 的 `transfer_to_x` 交接，而是：**supervisor 是一个普通 `create_agent`，每个子 Agent 被包成一个 `@tool`，工具内部 `sub_agent.invoke(...)` 同步执行并把结果作为 ToolMessage 返回**。子 Agent 的内部消息不会冒泡到父级，自然没有「孤儿 tool_call」问题。

## 四、修复方案（推荐顺序）

### 方案 A（首选，官方标准做法）：把 HITL 工具挂到 supervisor 层

**核心**：让 `select_origin` 这个 HITL 暂停发生在**顶层（supervisor）单一消息作用域**里，而不是子 Agent 里。前端的 `useHumanInTheLoop` 一行都不用改。

只需调整后端两处：

**1）把 `CopilotKitMiddleware()` 从子 Agent 移到 supervisor**

当前 `navigation_agent.py`（错误位置）：
```python
return create_agent(
    model=create_llm(temperature=0.3),
    tools=navigation_tools,
    name="navigation_agent",
    system_prompt=NAVIGATION_PROMPT,
    middleware=[CopilotKitMiddleware()],   # ← 删掉：HITL 不应发生在子 Agent
)
```
改为子 Agent 不带 `CopilotKitMiddleware`（只保留它自己的 MCP 工具：`plan_route` / `search_poi` / `traffic`）。

**2）supervisor 负责前端工具 + 起点询问**

`langgraph-supervisor` 的 `create_supervisor` 不接受 `middleware` 参数，所以要让 supervisor「看见」前端工具，有两种落地方式：

- **方式 A1（最小改动，仍用 langgraph-supervisor）**：supervisor 的 prompt 里读取 `state["copilotkit"]["actions"]`，把 `select_origin` 作为 supervisor 自己可调用的工具暴露出来——即在路由到 `navigation_agent` **之前**，由 supervisor 先调用 `select_origin` 询问起点，拿到结果后再把「从〈起点〉导航到〈终点〉」交给 navigation_agent。这样 HITL 的 tool_call 和 ToolMessage 都在 supervisor 的同一作用域内成对出现。

- **方式 A2（更彻底，对齐官方 Sub-Agents 写法）**：放弃 `langgraph-supervisor`，改成官方模式——
  - supervisor = `create_agent(..., middleware=[CopilotKitMiddleware()])`
  - 每个子 Agent 包成 `@tool`，内部 `sub_agent.invoke(...)` 同步返回 ToolMessage
  - `select_origin` / `update_map` 等前端工具天然由 supervisor 持有
  
  这种结构下，子 Agent 的消息不冒泡，HITL 暂停发生在顶层，`_validate_chat_history` 永远不会看到跨作用域的孤儿 tool_call。

> 方案 A2 是 CopilotKit 文档给出的「canonical multi-agent pattern」，最稳，但需要重写编排层（`supervisor.py` + 各 `*_agent.py` 的封装）。方案 A1 改动小，适合先快速止血。

---

### 方案 B（备选）：把 `select_origin` 做成「不暂停」的前端工具

如果业务上可以接受「不真正暂停 Agent」，可以把起点选择改成 `useFrontendTool`（像 `update_map` 那样同步返回），让前端 handler 弹出选择框、`await` 用户点击后 `return` 选择值。这样它就和 `update_map` 一样在同一 run 内闭环，不会留孤儿 tool_call，即使挂在子 Agent 上也不报错。

**代价**：失去 HITL 的「检查点暂停 / 可恢复」语义，本质上是把交互压进一次同步调用里。对「选个起点」这种轻量交互够用，但不如 A 方案规范。

---

### 关于 `useInterrupt` / LangGraph 原生 `interrupt()`

之前版本曾建议改用 LangGraph 原生 `interrupt()` + CopilotKit `useInterrupt`。**但需要注意**：CopilotKit 官方 HITL 文档明确标注——

> "**Not supported on LangGraph (Python)**: LangGraph (Python) doesn't support Human in the Loop: Interrupts."

即在 **LangGraph Python** 后端集成下，CopilotKit 的 `useInterrupt`（graph-paused 模式）当前**不在支持矩阵内**。虽然 LangGraph 本身有 `interrupt()` 原语，但要让它和 CopilotKit 前端打通需要走 headless 自定义方案，成本和不确定性都更高。**因此本项目应优先用方案 A（基于官方支持的 `useHumanInTheLoop`），而不是 `useInterrupt`。**

## 六、短期应急（仅救火，不是正解）

如果线上正在崩、来不及做架构调整，可以在子 Agent 调 LLM 前过滤掉孤儿 tool_call，**让校验不再炸**。但这只是把异常掩盖掉，HITL 的语义其实是断的（用户的选择不会被正确续接），**务必后续用方案 A 替换**。

```python
# app/agents/navigation_agent.py —— 应急 hack，非长期方案
from langchain_core.messages import AIMessage, ToolMessage, BaseMessage

def _drop_orphan_tool_calls(messages: list[BaseMessage]) -> list[BaseMessage]:
    """删除没有对应 ToolMessage 的孤儿 AIMessage tool_calls，避免 _validate_chat_history 抛错"""
    responded = {m.tool_call_id for m in messages if isinstance(m, ToolMessage)}
    out = []
    for msg in messages:
        if isinstance(msg, AIMessage) and msg.tool_calls:
            valid = [tc for tc in msg.tool_calls if tc["id"] in responded]
            if not valid:
                continue  # 整条都是孤儿，丢弃
            if len(valid) != len(msg.tool_calls):
                msg = msg.model_copy(update={"tool_calls": valid})
        out.append(msg)
    return out
```

> ⚠️ 该函数需要在子 Agent 真正调用 LLM 前执行（如 `pre_model_hook` / 自定义 `call_model`）。`langchain.agents.create_agent` 与 `langgraph.prebuilt.create_react_agent` 对钩子的支持不同，接入点需按实际版本验证。这只是止血，**不能替代方案 A**。
