## 1. 官方机制回顾（判定基准）

### 1.1 前端工具的官方注入链路（CopilotKit + AG-UI + LangGraph）

官方文档（CopilotKit — LangGraph FastAPI / Frontend Tools）描述的机制是：

```
前端 useFrontendTool / useHumanInTheLoop  ──(zod schema)──┐
                                                          │  请求体携带
                                                          ▼
                              AG-UI 请求体: state["copilotkit"]["actions"]
                                                          │
                                                          ▼
              后端 CopilotKitMiddleware.wrap_model_call：把 actions 合并进本次 model 调用的 tools
                                                          │
                                                          ▼
                          LLM 在本次调用中"看见"前端工具并决定调用
                                                          │
                                                          ▼
              after_model 钩子把前端工具调用从"后端待执行"列表摘出 → AG-UI 推到前端执行
```

**关键点：后端不需要为前端工具写任何 stub / StructuredTool / bind_tools。** 前端定义 schema，请求时带上，后端中间件动态注入。前端带了什么工具，LLM 这次就能看见什么——天然动态，无需重编图。

> 来源（项目内 2026-06-27 文档附录 A 已引用）：
> - CopilotKit — Frontend Tools：https://docs.copilotkit.ai/frontend-tools
> - CopilotKit — LangGraph FastAPI 前端工具（CopilotKitState / actions）：https://docs.copilotkit.ai/langgraph-fastapi/frontend-tools
> - copilotkit Python SDK 源码（CopilotKitMiddleware）：https://github.com/CopilotKit/CopilotKit/blob/main/sdk-python/copilotkit/copilotkit_lg_middleware.py

### 1.2 多 Agent 的官方写法（CopilotKit Sub-Agents + LangChain v1）

CopilotKit 官方 Sub-Agents 指南有一句关键注释（项目内 2026-06-28 排查文档已直接引用）：

> "We intentionally do **NOT** attach the full `CopilotKitMiddleware` here [on the sub-agents] — the supervisor handles App-context / frontend-tool injection for the whole run, and adding it on sub-agents would double-inject prompt state."

即官方推荐的多 Agent 写法是：

- **supervisor** = 普通 `create_agent(..., middleware=[CopilotKitMiddleware()])`，由它统一持有前端工具 / HITL；
- **每个子 Agent** 被包成一个 `@tool`，工具内部 `await sub_agent.ainvoke({"messages":[HumanMessage(task)]})` 同步执行，只把子 Agent 最终文本作为 ToolMessage 返回；
- 子 Agent 内部消息**不冒泡**到 supervisor 共享历史 → HITL 暂停只发生在 supervisor 单一作用域 → 不会产生跨作用域孤儿 tool_call（即不会触发 `INVALID_CHAT_HISTORY`）。
- 子 Agent **不挂** `CopilotKitMiddleware`，避免双重注入。

> 这正是项目 2026-06-28 排查文档中"方案 A2（更彻底，对齐官方 Sub-Agents 写法）"所采纳的模式。

### 1.3 LangChain v1 Agent API 基准

- `langgraph.prebuilt.create_react_agent` 已弃用（`LangGraphDeprecatedSinceV10`），官方提示 `use from langchain.agents import create_agent`。
- `create_agent(model, tools, *, system_prompt=…, middleware=[…], state_schema=…, checkpointer=…, name=…)` 是 v1 标准 harness。
- 参数名是 `system_prompt`（不是旧版 `prompt`）。
- 动态工具注入的官方扩展点 = `AgentMiddleware`（`wrap_model_call` / `before_model` / `after_model` / `wrap_tool_call`），`CopilotKitMiddleware`、`HumanInTheLoopMiddleware` 都是它的实例。

## 5. 给用户核心疑问的直接回答

> **"为啥必须把前端工具的规则写入后端中？不应该前端定义好，传输到后端吗？"**

**你说得对，官方就是这么做的不需要写进后端。** 当前 A2 架构的实际运行链路**已经是**"前端定义 → AG-UI 传输 → CopilotKitMiddleware 注入"，后端没有、也不需要为前端工具写 stub。问题在于仓库里还留着一份 `frontend_bridge_tools.py`，它在后端用 `StructuredTool` 重新定义了 `update_map` / `select_origin` 的 schema——这是 A2 迁移**之前**（A1 / langgraph-supervisor 时代）的遗留物，当时为了让后端 LLM"看见"前端工具才写的占位 stub。A2 之后这份文件已经无人引用，却仍躺在仓库里，制造"后端也要维护前端工具规则"的错觉，且与前端 zod、supervisor prompt 形成三份重复定义。**本次优化的核心动作就是删掉它，让"前端单一定义"成为唯一事实源。**
